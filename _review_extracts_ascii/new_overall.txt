城市道路突发事件应急抢修仿真与辅助决策系统
最终开发总纲 V2
本文档为 24 组“城市道路突发事件应急抢修仿真与辅助决策系统”的最终实施方案、前后端分工标准与固定接口契约。后端按本文档实现接口、业务规则和数据持久化；前端按本文档实现页面、交互和接口调用。本文档用于保证需求说明书、概要设计说明书、测试报告和最终代码实现口径一致，支持前后端并行开发。
一、项目最终定位
1. 项目名称
城市道路突发事件应急抢修仿真与辅助决策系统
2. 项目目标
本项目面向城市道路塌陷、暴雨积水、桥梁损坏、管线爆裂、普通事故占道等突发事件，构建一个可运行、可演示、可测试的 Web 系统。系统通过虚拟城市道路网络建模，模拟突发事件发生后的事件录入、资源调度、路径规划、方案生成、仿真推演和评估报告输出过程。
最终系统不是最低可行 Demo，而是完整覆盖 M1-M8 八个模块的课程项目：
M1 态势总览
M2 道路网络管理
M3 突发事件管理
M4 抢修资源管理
M5 路径规划
M6 方案生成
M7 仿真推演
M8 评估报告
3. 系统主流程
系统主流程如下：
加载虚拟路网和抢修资源↓创建道路突发事件↓事故路段状态变为 closed 或 damaged↓系统计算抢修队到事故点的可行路径↓生成最近队伍优先、专业能力优先、多队伍协同三类抢修方案↓按综合评分推荐最优方案↓启动仿真，更新队伍状态、道路状态和修复进度↓仿真完成后道路恢复，队伍恢复空闲↓输出评估报告并支持 JSON 导出
4. 不包含内容
本项目不做以下内容：
不接入真实高德/百度地图 API。
不做真实交通微观仿真。
不接入实时传感器或真实报警系统。
不做用户登录注册和权限管理。
不使用真实数据库。
不追求工业级最优算法。
这些内容作为后续扩展方向，不进入当前课程项目实现范围。
二、最终技术路线
1. 技术栈
最终采用：
前端：原生 HTML + CSS + JavaScript + SVG
后端：Python FastAPI
数据存储：本地 JSON 文件
部署方式：FastAPI 托管前端静态页面
运行方式：执行 python main.py 后访问 http://127.0.0.1:8000
2. 架构说明
系统采用“逻辑前后端分离、部署集成运行”的 B/S 架构。
前端页面由 FastAPI 静态托管，但前端仍通过 REST API 与后端交互。前端只负责展示、交互和接口调用；后端负责数据校验、业务规则、路径规划、方案评分、仿真推进和 JSON 文件持久化。
采用原生 HTML/CSS/JavaScript + SVG 的原因是降低 Node/Vite 等前端构建环境依赖，保证课程项目可以快速运行、演示和验收。
3. 后端必须拆分文件
后端不得将所有逻辑堆到 main.py 中。最终项目必须按职责拆分为：
main.py：FastAPI 启动、路由注册、静态资源托管。
models.py：Pydantic 模型、枚举、请求结构、响应结构。
storage.py：JSON 读写、初始化数据、原子写入、导入导出。
algorithms.py：Dijkstra、技能匹配、方案评分。
services.py：业务规则、状态联动、仿真推进、报告生成。
路由层只负责接收请求、调用 service、返回响应，不直接修改 JSON 文件。所有状态变更必须集中在 services.py 中完成。

四、统一接口规范
1. 成功响应
所有接口成功时统一返回：
{ “code”: 200, “message”: “success”, “data”: {} }
2. 失败响应
接口失败时统一返回：
{ “code”: 400, “message”: “错误说明”, “data”: null }
3. 错误码语义
200：成功。
400：参数错误。
404：对象不存在。
409：状态冲突或引用冲突。
500：服务端错误。
前端不直接依赖 HTTP 状态判断业务是否成功，前端统一判断 JSON 中的 code 字段，并展示 message。
五、数据结构标准
1. RoadNode 道路节点
示例：
{ “id”: “A”, “name”: “中心路口”, “x”: 120, “y”: 180, “type”: “intersection” }
字段说明：
id：节点编号，唯一字符串。
name：节点名称，不能为空。
x：SVG 横坐标，数字。
y：SVG 纵坐标，数字。
type：节点类型。
节点类型枚举：
intersection：普通路口。
station：抢修站。
depot：物资仓库点。
校验规则：
id 不能为空且唯一。
name 不能为空。
x 和 y 必须为数字。
type 必须为合法枚举。
2. RoadEdge 道路边
示例：
{ “id”: “AB”, “fromNode”: “A”, “toNode”: “B”, “length”: 1.2, “speed”: 40, “flow”: 900, “status”: “normal” }
字段说明：
id：道路编号，唯一字符串。
fromNode：起点节点 ID。
toNode：终点节点 ID。
length：道路长度。
speed：道路通行速度。
flow：交通流量。
status：道路状态。
道路状态枚举：
normal：正常。
congested：拥堵。
damaged：损坏。
closed：封闭。
repairing：修复中。
recovered：已恢复。
校验规则：
id 唯一。
fromNode 和 toNode 必须存在。
length > 0。
speed > 0。
flow >= 0。
status 必须为合法状态。
3. EmergencyEvent 突发事件
最终字段：
{ “id”: “E001”, “type”: “road_collapse”, “roadId”: “HI”, “severity”: “high”, “startTime”: “08:10”, “workload”: 120, “blocked”: true, “status”: “created”, “originalRoadStatus”: “normal”, “selectedPlanId”: null, “currentSimulationId”: null, “createdAt”: “2026-06-12 10:00:00”, “updatedAt”: “2026-06-12 10:00:00” }
字段说明：
id：事件 ID。
type：事件类型。
roadId：关联道路 ID。
severity：严重等级。
startTime：发生时间。
workload：抢修工作量。
blocked：是否封闭道路。
status：事件状态。
originalRoadStatus：事件创建前道路原始状态。
selectedPlanId：当前推荐或选中的方案 ID。
currentSimulationId：当前关联仿真 ID。
createdAt：创建时间。
updatedAt：更新时间。
事件类型枚举：
road_collapse：道路塌陷。
waterlogging：暴雨积水。
bridge_damage：桥梁损坏。
pipe_burst：管线爆裂。
traffic_accident：普通事故占道。
严重等级枚举：
low
medium
high
事件状态枚举：
created：已创建。
planned：已生成方案。
running：仿真中。
finished：已完成。
cancelled：已取消。
校验规则：
roadId 必须存在。
workload > 0。
type、severity、status 必须合法。
running 和 finished 状态事件不允许随意修改核心字段。
4. RepairTeam 抢修队
示例：
{ “id”: “T01”, “name”: “抢修一队”, “locationNode”: “A”, “workers”: 8, “vehicles”: 2, “skill”: “road_repair”, “efficiency”: 0.8, “status”: “idle” }
字段说明：
id：队伍 ID。
name：队伍名称。
locationNode：当前位置节点 ID。
workers：人员数。
vehicles：车辆数。
skill：专业能力。
efficiency：作业效率。
status：队伍状态。
技能枚举：
road_repair
drainage
bridge_repair
pipe_repair
general
队伍状态枚举：
idle：空闲。
busy：忙碌。
offline：离线。
校验规则：
locationNode 必须存在。
workers >= 0。
vehicles >= 0。
efficiency > 0。
skill 和 status 必须合法。
busy 队伍不允许删除。
被运行中仿真引用的队伍不允许删除。
5. MaterialDepot 物资仓库
示例：
{ “id”: “D01”, “nodeId”: “C”, “materialType”: “asphalt”, “stock”: 300 }
字段说明：
id：仓库 ID。
nodeId：仓库所在节点 ID。
materialType：物资类型。
stock：库存数量。
物资类型枚举：
asphalt
drainage_pump
steel
pipe
general
校验规则：
nodeId 必须存在。
stock >= 0。
materialType 必须合法。
6. RepairPlan 抢修方案
最终字段：
{ “id”: “P001”, “eventId”: “E001”, “strategy”: “nearest”, “strategyName”: “最近队伍优先”, “teams”: [“T01”], “route”: [“A”, “B”, “E”, “H”], “arrivalTime”: 12, “repairTime”: 80, “totalTime”: 92, “cost”: 3500, “affectedVehicles”: 1200, “riskPenalty”: 0.1, “score”: 0.65, “isRecommended”: false, “reason”: “到达速度快，但专业匹配度一般”, “createdAt”: “2026-06-12 10:20:00” }
字段说明：
id：方案 ID。
eventId：关联事件 ID。
strategy：策略编号。
strategyName：策略中文名称。
teams：参与队伍 ID 数组，只存队伍 ID，不存完整队伍对象。
route：路线节点 ID 数组。
arrivalTime：到达时间。
repairTime：修复时间。
totalTime：总恢复时间。
cost：成本。
affectedVehicles：影响车辆数。
riskPenalty：风险惩罚。
score：综合评分。
isRecommended：是否推荐。
reason：推荐或不推荐原因。
createdAt：创建时间。
策略枚举：
nearest：最近队伍优先。
skill_first：专业能力优先。
collaboration：多队伍协同。
评分规则：
Score = 0.45 × 归一化总恢复时间+ 0.25 × 归一化影响车辆数+ 0.20 × 归一化成本+ 0.10 × 风险惩罚
分数越低，推荐度越高。
7. Simulation 仿真状态
最终字段：
{ “id”: “SIM001”, “eventId”: “E001”, “planId”: “P003”, “status”: “running”, “currentTime”: 15, “progress”: 35, “speed”: 1, “roadId”: “HI”, “roadStatus”: “repairing”, “teamPositions”: { “T01”: “E”, “T03”: “F” }, “logs”: [ “第15分钟：抢修队正在前往事故路段” ], “beforeSimulationSnapshot”: { “eventStatus”: “planned”, “roadStatus”: “closed”, “teams”: [ { “id”: “T01”, “status”: “idle”, “locationNode”: “A” }, { “id”: “T03”, “status”: “idle”, “locationNode”: “F” } ] }, “startedAt”: “2026-06-12 10:30:00”, “finishedAt”: null }
字段说明：
id：仿真 ID。
eventId：关联事件 ID。
planId：采用的方案 ID。
status：仿真状态。
currentTime：当前仿真时间。
progress：修复进度，0-100。
speed：仿真倍速。
roadId：抢修道路 ID。
roadStatus：当前道路状态。
teamPositions：队伍当前位置映射。
logs：仿真日志。
beforeSimulationSnapshot：仿真开始前快照，用于 reset。
startedAt：开始时间。
finishedAt：完成时间。
仿真状态枚举：
ready
running
paused
finished
reset
六、状态联动规则
1. 创建事件
创建事件时：
校验 roadId 是否存在。
保存道路原状态到 event.originalRoadStatus。
event.status = created。
若 blocked = true，则关联道路 status = closed。
若 blocked = false，则关联道路 status = damaged。
写入 createdAt 和 updatedAt。
2. 编辑事件
编辑事件时：
created 和 planned 状态允许编辑。
running 和 finished 状态不允许修改核心字段。
如果 roadId 或 blocked 发生变化：
旧道路不能简单恢复 normal，必须调用道路状态重算函数。
新道路根据新事件状态更新为 closed 或 damaged。
更新时间 updatedAt。
3. 删除事件
删除事件时：
running 事件不允许删除。
created 和 planned 事件允许删除。
删除事件后，不能简单把道路恢复 normal。
必须调用 recalculate_road_status(roadId)，根据同一路段是否还有其他未完成事件重新计算道路状态。
4. 道路状态重算
必须在 services.py 中实现函数：
recalculate_road_status(road_id)
道路状态优先级：
如果该 roadId 有 running 事件，则道路状态为 repairing。
否则如果该 roadId 有 created/planned 且 blocked=true 的事件，则道路状态为 closed。
否则如果该 roadId 有 created/planned 且 blocked=false 的事件，则道路状态为 damaged。
否则如果该 roadId 有 finished 事件，则道路状态可保持 recovered。
否则恢复为事件记录的 originalRoadStatus 或 normal。
不得由单个事件删除或完成操作直接把道路恢复 normal。
5. 生成方案
生成方案时：
校验 eventId 存在。
事件状态必须为 created 或 planned。
只使用 status = idle 的队伍。
如果没有可用空闲队伍，返回 409。
同一 eventId 再次生成方案时，删除该 eventId 的旧方案，替换为新生成方案。
生成 nearest、skill_first、collaboration 三类方案。
评分最低方案 isRecommended = true。
其他方案 isRecommended = false。
event.status = planned。
event.selectedPlanId = 推荐方案 id。
更新 event.updatedAt。
6. 启动仿真
启动仿真时必须校验：
planId 存在。
plan.eventId 对应的 event 存在。
plan 必须属于该 event。
event.status 必须是 planned。
当前 event 不得已有 running simulation。
方案中的所有队伍必须存在。
方案中的所有队伍必须是 idle。
事故道路必须存在。
启动成功后：
创建 simulation。
保存 beforeSimulationSnapshot。
event.status = running。
event.currentSimulationId = simulation.id。
road.status = repairing。
方案中选中的队伍 status = busy。
simulation.status = running。
simulation.progress = 0。
simulation.currentTime = 0。
写入开始日志。
7. 暂停仿真
暂停仿真时：
simulation.status 必须是 running。
设置 simulation.status = paused。
暂停后 step 不推进。
8. 继续仿真
继续仿真时：
simulation.status 必须是 paused。
设置 simulation.status = running。
9. 推进一步仿真
推进接口使用 POST，不使用 GET。
请求体：
{ “simulationId”: “SIM001” }
推进规则：
如果 simulation.status 不是 running，不推进，只返回当前状态。
每步 currentTime += 5 × speed。
根据方案 totalTime 推进 progress。
更新 teamPositions。
追加日志。
progress 不超过 100。
当 progress >= 100 时：
simulation.status = finished。
simulation.progress = 100。
event.status = finished。
road.status = recovered。
方案中队伍 status = idle。
方案中队伍 locationNode = 事故道路 fromNode 或 toNode。
simulation.finishedAt = 当前时间。
写入完成日志。
10. 重置仿真
重置仿真时必须使用 beforeSimulationSnapshot 恢复状态：
simulation.status = reset。
simulation.progress = 0。
simulation.currentTime = 0。
event.status 恢复为快照中的 eventStatus，通常为 planned。
road.status 恢复为快照中的 roadStatus，通常为 closed 或 damaged。
队伍恢复为快照中的 status 和 locationNode。
event.currentSimulationId 可置空或保留最近 simulationId，但前端以后端返回为准。
追加重置日志。
七、最终接口契约
所有接口统一返回：
{ “code”: 200, “message”: “success”, “data”: {} }
1. 系统状态接口
GET /api/state
用途：前端初始化和刷新全局状态。
返回：
{ “code”: 200, “message”: “success”, “data”: { “nodes”: [], “edges”: [], “events”: [], “teams”: [], “depots”: [], “plans”: [], “simulations”: [], “metrics”: { “eventCount”: 0, “activeTasks”: 0, “averageRecoveryTime”: 0, “affectedVehicles”: 0, “recoveredRoads”: 0 } } }
POST /api/reset-data
用途：重置演示数据。
请求：
{}
返回：
{ “code”: 200, “message”: “演示数据已重置”, “data”: true }
M2 道路网络管理接口
GET /api/network
返回完整路网：
{ “code”: 200, “message”: “success”, “data”: { “nodes”: [], “edges”: [] } }
GET /api/nodes
返回节点列表。
POST /api/nodes
请求：
{ “id”: “J”, “name”: “新路口”, “x”: 300, “y”: 240, “type”: “intersection” }
失败情况：
id 重复：409。
字段缺失：400。
坐标非法：400。
类型非法：400。
PUT /api/nodes/{id}
请求：
{ “name”: “修改后的路口”, “x”: 320, “y”: 250, “type”: “intersection” }
说明：不允许修改节点 id。
DELETE /api/nodes/{id}
规则：
如果节点被以下对象引用，则拒绝删除：
任意 edge.fromNode。
任意 edge.toNode。
任意 team.locationNode。
任意 depot.nodeId。
失败返回：
{ “code”: 409, “message”: “节点正在被道路、队伍或仓库引用，无法删除”, “data”: null }
GET /api/edges
返回道路列表。
POST /api/edges
请求：
{ “id”: “JK”, “fromNode”: “J”, “toNode”: “K”, “length”: 1.5, “speed”: 40, “flow”: 600, “status”: “normal” }
校验：
id 不重复。
fromNode/toNode 存在。
length > 0。
speed > 0。
flow >= 0。
status 合法。
PUT /api/edges/{id}
请求：
{ “fromNode”: “J”, “toNode”: “K”, “length”: 1.8, “speed”: 35, “flow”: 700, “status”: “congested” }
说明：允许修改道路状态，用于手动制造拥堵、封闭、恢复等场景。
DELETE /api/edges/{id}
规则：
如果道路被未完成事件引用，则拒绝删除。
未完成事件包括：
created
planned
running
失败返回：
{ “code”: 409, “message”: “道路正在被未完成事件引用，无法删除”, “data”: null }
GET /api/network/export
返回：

3. Dijkstra 算法标准
路径权重：
权重 = length / speed × congestionFactor
权重规则：
normal/recovered：congestionFactor = 1。
congested：congestionFactor = 1.5。
closed/damaged/repairing：不可通行。
返回字段：
path
cost
distance
blockedEdgesSkipped
异常：
起点不存在，返回 400。
终点不存在，返回 400。
不可达，返回业务错误提示。
4. 方案生成标准
必须生成三类方案：
nearest：最近队伍优先。
skill_first：专业能力优先。
collaboration：多队伍协同。
技能匹配规则：
road_collapse → road_repair
waterlogging → drainage
bridge_damage → bridge_repair
pipe_burst → pipe_repair
traffic_accident → general 或 road_repair
修复时间：
单队伍：
repairTime = workload / efficiency
多队伍协同：
repairTime = workload / (efficiency1 + efficiency2) × 0.85
协同方案特点：
修复时间缩短。
成本增加。
影响车辆数降低。
评分规则：
Score = 0.45 × 归一化总恢复时间+ 0.25 × 归一化影响车辆数+ 0.20 × 归一化成本+ 0.10 × 风险惩罚
分数最低方案为推荐方案。
