五、路网管理规则
节点新增
必须校验：
id 唯一name 非空x/y 是数字type 合法
节点删除
如果节点被以下对象引用，拒绝删除：
edge.fromNodeedge.toNodeteam.locationNodedepot.nodeId
返回 409。
道路新增
必须校验：
id 唯一fromNode 存在toNode 存在length > 0speed > 0flow >= 0status 合法
道路删除
如果道路被未完成事件引用，拒绝删除。
未完成事件：
createdplannedrunning
返回 409。
network 导入
必须先完整校验：
nodes 存在edges 存在节点 id 不重复道路 id 不重复道路 fromNode/toNode 都存在数值字段合法
校验失败不能覆盖旧数据。
六、事件管理规则
创建事件
请求字段：
typeroadIdseveritystartTimeworkloadblocked
规则：
1. roadId 必须存在。2. 保存 originalRoadStatus。3. event.status = created。4. blocked=true 时 road.status=closed。5. blocked=false 时 road.status=damaged。6. selectedPlanId = null。7. currentSimulationId = null。8. 写 createdAt、updatedAt。
编辑事件
规则：
1. created/planned 可以编辑。2. running/finished 限制编辑核心字段。3. 如果 roadId 或 blocked 改变：   - 旧道路调用 recalculate_road_status。   - 新道路根据 blocked 设为 closed 或 damaged。4. 更新 updatedAt。
删除事件
规则：
1. running 事件不能删除。2. 删除后调用 recalculate_road_status。3. 不能简单把道路恢复 normal。
道路状态重算
必须实现：
recalculate_road_status(road_id)
规则：
优先级：1. 如果同 roadId 有 running 事件，road.status=repairing。2. 否则如果有 created/planned 且 blocked=true 事件，road.status=closed。3. 否则如果有 created/planned 且 blocked=false 事件，road.status=damaged。4. 否则如果有 finished 事件，可保持 recovered。5. 否则恢复 originalRoadStatus 或 normal。

八、路径规划规则
接口：
GET /api/path?from=A&to=I
算法：
Dijkstra 加权最短路径
权重：
weight = length / speed × congestionFactor
状态规则：
normal/recovered: factor=1congested: factor=1.5closed/damaged/repairing: 不可通行
返回：
{  "from": "A",  "to": "I",  "path": ["A", "B", "E", "H", "I"],  "cost": 18.5,  "distance": 7.2,  "blockedEdgesSkipped": ["HI"]}
异常：
起点不存在：400终点不存在：400不可达：400
九、方案生成规则
接口：
POST /api/plans/generate
请求：
{  "eventId": "E001"}
规则：
1. eventId 必须存在。2. event.status 必须是 created 或 planned。3. 只使用 idle 队伍。4. 没有 idle 队伍，返回 409。5. 同一事件重新生成方案时，先删除该事件旧 plans。6. 每次只保留该事件最新 3 个方案。7. 生成 nearest、skill_first、collaboration。8. 计算评分。9. 分数最低 isRecommended=true。10. event.selectedPlanId=推荐方案 id。11. event.status=planned。12. 更新 event.updatedAt。
三种策略
nearest
选择到达时间最短的 idle 队伍。
skill_first
选择专业能力匹配的 idle 队伍，再按到达时间排序。
技能匹配：
road_collapse -> road_repairwaterlogging -> drainagebridge_damage -> bridge_repairpipe_burst -> pipe_repairtraffic_accident -> general 或 road_repair
collaboration
选择两支较优 idle 队伍协同。
如果 idle 队伍不足两支：
可以返回单队伍协同降级方案，或返回无法生成协同方案的 reason。但前端仍应收到三类方案中的可展示结果。
修复时间
单队伍：
repairTime = workload / efficiency
多队伍：
repairTime = workload / (efficiency1 + efficiency2) × 0.85
评分
Score = 0.45 × 归一化总恢复时间      + 0.25 × 归一化影响车辆数      + 0.20 × 归一化成本      + 0.10 × 风险惩罚
分数越低越推荐。
十、仿真规则
启动仿真
接口：
POST /api/simulation/start
请求固定为：
{  "eventId": "E001",  "planId": "P003"}
启动前必须校验：
1. eventId 存在。2. planId 存在。3. planId 属于该 eventId。4. event.status 必须是 planned。5. 同一事件不能存在 running 或 paused 仿真。6. 方案中的所有队伍存在。7. 方案中的所有队伍 status=idle。8. 事故 roadId 存在。
启动后：
1. 创建 simulation。2. 保存 beforeSimulationSnapshot。3. event.status=running。4. event.currentSimulationId=simulation.id。5. road.status=repairing。6. 方案中 teams 的 status=busy。7. simulation.status=running。8. 写入 startedAt。
推进一步
接口：
POST /api/simulation/step
请求：
{  "simulationId": "SIM001"}
规则：
1. 只认 simulationId。2. 如果 simulation.status != running，不推进。3. currentTime += 5 × speed。4. progress 按 totalTime 推进。5. progress >= 100 时完成仿真。
完成后：
simulation.status=finishedevent.status=finishedroad.status=recoveredteams.status=idleteams.locationNode=事故道路 fromNode 或 toNodefinishedAt=当前时间
暂停
接口：
POST /api/simulation/pause
请求：
{  "simulationId": "SIM001"}
规则：
只有 running 可以暂停。暂停后 status=paused。
继续
接口：
POST /api/simulation/resume
请求：
{  "simulationId": "SIM001"}
规则：
只有 paused 可以继续。继续后 status=running。
重置
接口：
POST /api/simulation/reset
请求：
{  "simulationId": "SIM001"}
规则：
1. 必须使用 beforeSimulationSnapshot。2. event.status 恢复 planned。3. road.status 恢复为仿真前状态。4. teams 恢复仿真前 status 和 locationNode。5. simulation.status=reset。6. progress=0。7. currentTime=0。
查询仿真
接口：
GET /api/simulation/{simulationId}
用途：
前端刷新页面恢复仿真报告读取仿真日志查看当前状态

十二、后端验收标准
后端完成后必须满足：
1. python main.py 能启动。2. 首页能访问。3. 所有接口返回 {code,message,data}。4. CRUD 操作真实写入 JSON。5. JSON 写入使用原子写。6. JSON 导入失败不覆盖原数据。7. 删除被引用节点会返回 409。8. 删除被事件引用道路会返回 409。9. running 事件不能删除。10. busy 队伍不能删除。11. Dijkstra 能绕开 closed/damaged/repairing。12. 所有队伍 busy 时生成方案返回 409。13. 同一事件重新生成方案会覆盖旧方案。14. 启动仿真会校验 eventId + planId。15. 同一事件不能同时有 running/paused 仿真。16. step 是 POST。17. reset 能按 beforeSimulationSnapshot 恢复。18. 报告能聚合事件、方案、仿真日志。
