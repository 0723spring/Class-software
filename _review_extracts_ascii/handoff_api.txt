前后端接口交接说明
本文档用于前端和后端并行开发时统一接口、字段和状态规则。前端只负责页面展示和接口调用，后端负责业务逻辑、数据校验、路径规划、方案评分、仿真推进和 JSON 持久化。双方不得自行修改接口路径、字段名和状态枚举。
一、统一响应格式
所有接口必须统一返回：
{  "code": 200,  "message": "success",  "data": {}}
失败时返回：
{  "code": 400,  "message": "错误说明",  "data": null}
错误码语义固定：
200：成功400：参数错误404：对象不存在409：状态冲突或引用冲突500：服务端错误
前端统一判断 code，并展示 message。
二、前后端固定字段名
以下字段名必须完全一致，不允许改成下划线或其他写法：
idfromNodetoNodelocationNodenodeIdroadIdeventIdplanIdsimulationIdselectedPlanIdcurrentSimulationIdarrivalTimerepairTimetotalTimeaffectedVehiclesriskPenaltyisRecommendedbeforeSimulationSnapshotcreatedAtupdatedAtstartedAtfinishedAt
禁止出现前端写 from_node、后端写 fromNode 这种情况。

五、重点接口说明
1. GET /api/state
用途：前端初始化和每次操作成功后的全局刷新。
返回：
{  "code": 200,  "message": "success",  "data": {    "nodes": [],    "edges": [],    "events": [],    "teams": [],    "depots": [],    "plans": [],    "simulations": [],    "metrics": {      "eventCount": 0,      "activeTasks": 0,      "averageRecoveryTime": 0,      "affectedVehicles": 0,      "recoveredRoads": 0    }  }}
2. POST /api/events
请求：
{  "type": "road_collapse",  "roadId": "HI",  "severity": "high",  "startTime": "08:10",  "workload": 120,  "blocked": true}
后端规则：
1. roadId 必须存在。2. 创建 event，状态为 created。3. 保存 originalRoadStatus。4. blocked=true 时道路变 closed。5. blocked=false 时道路变 damaged。6. 写入 createdAt、updatedAt。
3. PUT /api/events/{id}
规则：
1. created/planned 状态可以编辑。2. running/finished 状态不允许修改核心字段。3. 如果 roadId 或 blocked 改变，旧道路必须重新计算状态，新道路按事件状态更新。
4. DELETE /api/events/{id}
规则：
1. running 事件不允许删除。2. 删除事件后不得简单把道路恢复 normal。3. 必须根据同一路段是否还有其他未完成事件重新计算道路状态。
5. POST /api/plans/generate
请求：
{  "eventId": "E001"}
硬规则：
1. eventId 必须存在。2. 事件状态必须是 created 或 planned。3. 只使用 status=idle 的队伍。4. 如果没有 idle 队伍，返回 409。5. 同一事件重新生成方案时，先删除该 eventId 的旧 plans。6. 每次只保留当前事件最新的 3 个方案。7. 新生成 nearest、skill_first、collaboration 三类方案。8. score 最低的方案 isRecommended=true。9. 同步 event.selectedPlanId = 推荐方案 id。10. 同步 event.status = planned。
返回：
{  "code": 200,  "message": "success",  "data": [    {      "id": "P001",      "eventId": "E001",      "strategy": "nearest",      "strategyName": "最近队伍优先",      "teams": ["T01"],      "route": ["A", "B", "E", "H"],      "arrivalTime": 12,      "repairTime": 80,      "totalTime": 92,      "cost": 3500,      "affectedVehicles": 1200,      "riskPenalty": 0.1,      "score": 0.65,      "isRecommended": false,      "reason": "到达速度快，但专业匹配度一般",      "createdAt": "2026-06-12 10:20:00"    }  ]}
6. POST /api/simulation/start
前后端固定：启动仿真时传 eventId + planId。
请求：
{  "eventId": "E001",  "planId": "P003"}
后端必须校验：
1. eventId 必须存在。2. planId 必须存在。3. planId 必须属于该 eventId。4. event.status 必须是 planned。5. 同一事件不能存在另一个 running 或 paused 仿真。6. 方案中的所有队伍必须存在。7. 方案中的所有队伍必须是 idle。8. 事故 roadId 必须存在。
启动成功后：
1. 创建 simulation。2. 保存 beforeSimulationSnapshot。3. event.status = running。4. event.currentSimulationId = simulation.id。5. road.status = repairing。6. 方案中队伍 status = busy。7. simulation.status = running。
返回 simulation 对象。
7. POST /api/simulation/step
推进仿真只认 simulationId。
请求：
{  "simulationId": "SIM001"}
规则：
1. 如果 simulation.status != running，不推进，只返回当前状态。2. 每步 currentTime += 5 × speed。3. progress 按 totalTime 推进。4. progress >= 100 时完成仿真。5. 完成后 event.status=finished，road.status=recovered，teams.status=idle。
8. POST /api/simulation/pause
请求：
{  "simulationId": "SIM001"}
规则：
只有 running 状态可以 pause。pause 后 simulation.status=paused。
9. POST /api/simulation/resume
请求：
{  "simulationId": "SIM001"}
规则：
只有 paused 状态可以 resume。resume 后 simulation.status=running。
10. POST /api/simulation/reset
请求：
{  "simulationId": "SIM001"}
规则：
1. 必须使用 beforeSimulationSnapshot 恢复状态。2. event.status 恢复为 planned。3. road.status 恢复为仿真前状态。4. teams 恢复为仿真前 status 和 locationNode。5. simulation.status=reset。6. progress=0。7. currentTime=0。
11. GET /api/simulation/{simulationId}
用途：
1. 前端刷新页面后恢复仿真状态。2. 报告页读取仿真日志。3. 查看当前 progress、status、teamPositions。
12. GET /api/report/{eventId}/export
返回格式仍然是统一结构：
{  "code": 200,  "message": "success",  "data": {    "event": {},    "recommendedPlan": {},    "plans": [],    "simulation": {},    "summary": {}  }}
注意：
后端不直接返回文件流。前端基于 data 自己生成 report-E001.json 并下载。
六、前端刷新规则
以下操作成功后，前端必须重新调用 GET /api/state：
创建事件编辑事件删除事件加载预设场景新增/编辑/删除节点新增/编辑/删除道路导入 network新增/编辑/删除队伍新增/编辑/删除仓库导入 resources生成方案启动仿真推进仿真暂停仿真继续仿真重置仿真
七、前端禁止事项
前端不得自己计算：
路径规划方案评分推荐方案仿真进度道路状态恢复队伍 busy/idle
这些全部以后端返回为准。
