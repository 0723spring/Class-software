# 开发记录与已处理问题

本文档记录后端第一轮开发过程中遇到的关键问题、处理方式和后续建议，便于课程总结与答辩复盘。

## 1. 环境问题

- `pytorch_env` 初始缺少 `fastapi`
  - 处理：安装 `fastapi`
  - 影响：未安装前无法启动后端服务

- `uvicorn` 依赖的 `click` 包损坏
  - 现象：导入 `uvicorn` 时出现 `module 'click' has no attribute 'Choice'`
  - 排查结果：`site-packages/click/` 目录存在，但缺少 `__init__.py`
  - 处理：强制重装 `click`
  - 影响：未修复前 `python main.py` 无法启动

## 2. 后端实现问题

- 统一响应包装最初未生效
  - 现象：接口返回的不是统一 `{code, message, data}` 结构
  - 原因：装饰器未保留原函数元信息，导致 FastAPI 路由解析异常
  - 处理：在 `handle_service` 中增加 `@wraps(func)`

- 方案生成时 `planId` 有撞车风险
  - 现象：同一次生成的三个方案可能出现相同 `planId`
  - 风险：前端选择某个方案后，后端可能实际启动成另一方案
  - 处理：重写方案 ID 分配逻辑，保证单次生成的三个方案 ID 唯一

## 3. 已验证主流程

已跑通以下链路：

- `GET /api/state`
- `POST /api/events`
- `POST /api/plans/generate`
- `POST /api/simulation/start`
- `POST /api/simulation/step`
- `GET /api/report/{eventId}`

已验证以下异常：

- `running` 事件不可删除
- `busy` 队伍不可删除

## 4. 当前已知限制

- 第一轮前端页面只提供了最小静态页，用于确认 FastAPI 托管正常
- 仿真日志目前是基础版，可在第二轮继续细化
- 报告页目前是后端聚合数据，前端展示样式仍待接入

## 5. 第二轮建议

- 补充前端六个 Tab 的正式页面骨架
- 细化导入导出校验和错误提示
- 丰富仿真日志阶段信息
- 增加更多异常测试与联调测试记录

## 6. 第二轮已补强问题

- 协同方案不再天然占优
  - 处理：方案生成时按事件类型计算队伍技能匹配系数，非匹配队伍只提供有限效率贡献
  - 结果：`skill_first`、`nearest`、`collaboration` 会根据事件类型和队伍组合动态竞争，不再总是推荐最后一张方案卡

- 仿真恢复效果改为渐进式
  - 处理：仿真过程新增 `phase`、`currentAffectedVehicles`、`teamPositions`
  - 结果：前端可按步骤展示队伍移动、道路修复过程、影响车辆数下降

- 物资正式进入业务链路
  - 处理：方案阶段返回 `requiredMaterials`，仿真启动时校验库存并扣减，重置仿真时回滚库存
  - 结果：库存不足会阻止仿真启动，报告可展示消耗材料

- 支持多事件并行管理
  - 处理：允许多个事件同时存在，并允许最多 10 个 `running/paused` 仿真并行
  - 限制：同一事件同一时刻只允许一个活动仿真；同一队伍被占用后不可重复分配

## 7. 本轮联调中发现的问题与修正

- 问题：方案卡片看起来总是最后一个方案被推荐
  - 原因：协同方案原先在修复时间上优势过大，技能不匹配惩罚不足，前端又容易默认把最后一张卡当成“更优”
  - 修正：
    - 后端引入技能匹配效率折减，非匹配队伍效率贡献下降
    - 协同方案增加一定协同折损，不再无脑最优
    - 前端交接中明确要求按 `isRecommended` 判断推荐，并标注“`score` 越低越优”

- 问题：仿真推进后，道路与指标变化不明显，像“卡住不恢复”
  - 原因：之前仿真主要追加日志，缺少阶段字段和动态指标
  - 修正：
    - 后端增加 `phase`
    - 后端增加 `currentAffectedVehicles` 的渐进下降逻辑
    - 后端增加 `teamPositions`，便于前端做 SVG 动画
    - 仿真启动后道路立即进入 `repairing`，避免界面误以为未开始

- 问题：物资仓库存在，但与仿真流程脱节
  - 原因：原先只有资源展示，没有进入方案与仿真约束
  - 修正：
    - 新增材料需求估算规则
    - 方案返回 `requiredMaterials`
    - 方案返回 `materialFeasible`、`materialShortage`
    - 启动仿真时真实扣减库存
    - 重置仿真时恢复库存，便于测试与反复演示

- 问题：系统需要支持多个事件，但又不能做成很重的并发系统
  - 原因：课程项目需要“多事件可管理”，但实现上不能无限并发
  - 修正：
    - 允许多个事件同时存在
    - 允许最多 10 个 `running/paused` 仿真
    - 同一事件只允许一个活动仿真
    - 同一队伍忙碌时不能被重复分配
  - 说明：这样既能支撑测试与答辩，又不会把系统复杂度拉得过高

- 问题：前后端容易对“方案评分”理解不一致
  - 原因：分数设计为“越低越优”，但前端常见进度条/评分条习惯是“越高越好”
  - 修正：
    - 在前端交接文档中单独强调“综合评分越低越优”
    - 要求前端推荐高亮完全以后端 `isRecommended` 为准

- 问题：前后端联调时容易出现重复提交、重复推进仿真
  - 原因：生成方案、开始仿真、自动步进都属于状态型操作
  - 修正：
    - 前端交接中要求所有提交类操作加 `isSubmitting`
    - 仿真运行时由前端单一定时器轮询 `POST /api/simulation/step`
    - 暂停时停止轮询，继续时恢复轮询，避免多重定时器叠加

- 问题：导入路网/资源文件可能破坏现有方案或仿真
  - 原因：如果导入后节点、道路、队伍消失，运行中的方案与仿真会失效
  - 修正：
    - 网络导入前校验事件 `roadId`、方案路径、仿真队伍位置和快照节点
    - 资源导入前校验方案引用队伍与运行中仿真引用队伍
    - 校验失败直接返回错误，禁止覆盖原数据

- 问题：测试如果只测主流程，答辩时容易被老师问倒
  - 原因：课程项目更看重“异常处理”和“状态联动”
  - 修正：
    - 烟雾测试新增库存扣减/回滚校验
    - 新增推荐方案正确性校验
    - 新增仿真推进后影响车辆下降校验
    - 新增多仿真并行校验
    - 新增导入保护、运行中删除保护等异常校验
